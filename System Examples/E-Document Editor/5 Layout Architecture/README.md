# Модель 5-уровневой архитектуры редактора электронных документов

**Список сокращений:**
1. «Core System» = Система - совокупность процессов ОС и специфики обмена данными между ними, создающих «систему редактирования электронных документов».
0. «5-Level Core Model» = Модель - модель любого рода, описывающая один или более компонентов Системы, с одного или одновременно нескольких представлений.
0. «DOM» = Document Object Model - объектна модель электронного документа в памяти компьютера (специальная  COM like структура данных).
0. «Main Editor» = Редактор - компонента Модели, ответственная за визуализацию структур и прямое взаимодействие с пользователем.
0. Reserch and Development Department = R&D = ДР - организационно штатная часть Компании, ответственная за разработку Системы.
0. Test Department = DT = ДТ - Департамент Тестирования - организационно-штатная часть Компании, ответственная за тестирование Системы.

# Задача

Необходимо создать редактор документов, обладающий универсальной гибкостью адаптации к:
1) работе с документами любого рода:
    Редактор не должен налагать какие-либо ограничения на возможность работы с документами любых типов (текст, таблицы, презентации, картинки, музыка, и т.д.), но предоставлять возможности настройки для работы с ними.
2) настройке пользовательского интерфейса:
    Редактор не должен обязывать любых своих пользователей привыкать к своему рабочему интерфейсу, но должен предоставлять возможность гибкой настройки последнего в согласии с пожеланиями пользователей.
3) требованиям B&B и B&C клиентов одновременно:
    Редактор (в контексте всей Системы) должен быть универсальной платформой для решения самых необычных задач,как рядовых пользователей так и бизнес-пользователей, в том смысле, что решение задачи в Редакторе - это, во-первых, процесс создания рабочей среды (настройка Редактора на визуальном и структурном уровнях), и, во-вторых, собственно, процесс работы с целевым документом.

# НеФункциональные требования (DEV-специфичные принципы и положения)
## Экспансивное развитие
1) Архитектура должны иметь максимальную (избыточную) гибкость и минимальную связность!
    - изначально должна закладываться возможность неограниченного расширения в смысле:
        - работы с новыми стандартами документов;
        - поддержки новых объектов документа и/или способов работы с ними.
2) Архитектура должна быть спроектирована с учётом возможности допускать ошибки (допускать много ошибок)!
    - никакая бизнес-логика не должна быть краеугольным камнем бизнеса, но только структуры данных:
        - структуры данных (главным образом, DOM) должны быть:
            - исключительно простыми;
            - легко документируемыми;
            - не иметь специализированного предназначения, но быть быть общим инструментом описания;
        - бизнес-логика должна быть:
            - приближенной к доменному типу (быть слабо связанной, либо вообще никак не связанной с любой другой бизнес-логикой);
            - являться не более чем компонентом общего приложения, и никогда его структурной частью;
            - поощрять ситуацию переделывания/переписывания соответствующего кода, а не его исправления, в случае обнаружения несовершенств;
            - версионируемой в тот смысле, что определённый функционал должен быть доступен к использованию в любой из своих релизных версий.

## Один компонент - один цикл!
1) Каждый компонент Системы должен быть изолирован от других компонент в смысле своего жизненного цикла:
    - компоненты должны быть насколько маленькими насколько позволяет сфера их ответственности!
    - сфера ответственность каждого компонента не должна пересекаться со сферой ответственности любого другого компонента:
        - каждый компонент должен:
            - обладать обособленным циклом релизов и обновлений;
            - обладать независимой документацией;
            - быть независимой единицей кода:
                - иметь собственный git-репозиторий;
                - быть независимым по стандартам кода (возможно даже по стандартам языка);
                - не иметь связей с любыми другими компонентами Системы **одного с собой уровня или выше**:
                    - каждый компонент должен быть полностью независимым на этапе компиляции от других компонентов Системы;
                    - каждый компонент должен быть полностью независимым на этапе связывания от других компонентов Системы;
                    - тестирование и разворачивание компонента не должно зависеть от любого другого компонента Системы;
            - быть свободным от ситуации, когда бы ошибки разработки в одной команде приводили к простою/ошибкам в другой команде;

## Одна команда - один мир!
1) Каждая команда несёт ответственность только за какой-то один (один единственных) компонент Системы и его жизненный цикл:
    - должны быть обеспечены условия для комфортного (естественного) соблюдения нижеследующих положений:
        - сложность компонента не должна превышать порога когда бы требовалась команда более чем из 10 человек:
            - каждая команда обязуется вести полноценную документацию того компонента за который ответственна и таковая документация должна быть частью подответственной им кодовой базы:
                - если какие-то изменения в коде компонента подразумевают изменения с архитектуре/структуре/логике данного компонента, то соответствующий ПР должен содержать изменения в документации;
2) Каждая команда должна быть полностью независимой от любой другой команды:
    - никакие команды из разработки не должны быть вынуждены разделять (подстраиваться/ожидать/вникать в детали/согласовывать) между собой:
        - кодовую базу (могу быть только разделяемые библиотеки);
        - график релизов/обновлений;
        - стандарты кодирования, техники написания кода:
        - правила unit-тестирования;
3) Никакая команда не должна быть ответственной за проблемы возникающие на этапе интеграционного тестирования с нижележащими по уровню компонентами.
4) Никакая команда не должна быть ответственной за проблемы возникающие на этапе отладочного тестирования с вышележащими по уровню компонентами.

## Документов много - редактор один!
1) Принцип одного окна:
    - у пользователей должна отпасть необходимость работать с разными окнами в условиях многозадачности по документами, но должна появиться возможность работать с разными вкладками одного Редактора.
2) Принцип: Редактор - это платформа!
    - Система не уровне Редактора не должна делать предположения относительно желательного пользовательского опыта.
        - Система должна предоставлять инструменты настройки/адаптации Редактора (каждой отдельной вкладки редактора), для работы пользователя, относительно:
            - типа редактируемого документа
            - используемого для редактирования плагина
                - и только конкретный Плагин, как компонент Системы, производя настройку Редактора, делает предположение относительно желаемого пользовательского опыта:
                    - количество, тип и расположение элементов управления
                    - доступный функционал редактирования
                    - и т.д...

## Test Department ≠ R&D Department
Не должно быть ситуации, когда:
    - ДР делает что-то только для ДТ
    - ДТ зависит от ДР в принципах и процессах тестирования
    - Циклы работы этих департаментов имеют иную связь кроме релизной

## Open Source Community
1) Компания даёт (продаёт) платформу, а не функционал:
    - нужно дать возможность любым пользователям создавать собственные надстройки-плагины над основным приложением:
        - компания владеет только ядром, но не владеет интересами его использования
        - избыточная клиентоориентированность:
            - каждая компания должна иметь возможность создать сама или заказать у Компании плагин, специфичный только в условиях её бизнеса
2) Компания, как DEV-маркет:
    - Компания должна создать Систему, которая в будущем сама-собой станет экосистемой в масштабах одного приложения:
        - услуги Компании будут содержаться ещё и в том, чтобы по заказу сторонних организаций, создавать для них приватные-плагины, для их нужд, и осуществлять процесс их поддержки.
        - одновременно с этим, Компания может организовать магазин плагинов внутри своей Системы, и получать комиссию с каждой покупки, совершаемой между её пользователями.
