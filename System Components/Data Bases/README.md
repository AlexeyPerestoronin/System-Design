# Пояснения

**ACID** - набор требований к транзакционной системе работы с данными (СУБД):
* **Atomicity**:  
   Транзакция выполняется как единое целое — либо полностью, либо не выполняется вообще.  
   Нет промежуточных состояний.  
* **Consistency**:  
   Транзакция переводит базу данных из одного корректного состояния в другое корректное состояние.  
   Все правила, ограничения и инварианты сохраняются.  
* **Isolation**:  
   Параллельные транзакции не должны влиять друг на друга.  
   Результат параллельного выполнения должен быть эквивалентен последовательному.  
* **Durability**:  
   После успешного завершения транзакции (COMMIT) изменения сохраняются навсегда, даже при сбоях системы.  
   Имеется возможность гарантированного отката системы к прошлым состояниям.  

---

**CAP** - акроним теоремы Брюера - теорема о том, что в любой реализации распределённых вычислений (распределённой системы обработки данных РСОД) возможно обеспечить не более двух из трех следующих свойств:
* **Consistency** - когда данные во всех частях РСОД не противоречат друг другу
* **Availabililty** - когда запрос на доступ к данным РСОД и меет мгновенный отклик
* **Partition Tolerance** - расщепление РСОД не приводит к некорректности отклика

## О проблеме репликации данных

Реляционные БД, при использовании в промышленных масштабах, часто сталкиваются с проблемами повышения производительности и доступности, каковые решаются репликацией.

Техники репликации:
1. **Master-Master Replication** - техника репликации при которой все сервера имеют права добавлять данные в разделяемую БД и постоянно осуществлять процесс синхронизации.
2. **Master-Slave Replication** - техника реплицирования при которой только один сервер обладает правами записи в разделяемую БД, а все остальные могут только читать → запись происходит в однопоточном режиме, а чтение в многопоточном.
3. **Federation** - техника разделения БД по функциям ~ ситуация, когда единая SQL-схема разделяется на несколько максимально независимых, каждая из которых передаётся под управление отдельному серверу.
4. **Sharding** - техника разделения БД по срезам ~ ситуация, когда единый объём хранимых данных разделяется на некоторое количество максимально независимых срезов данных, каждый из которых передаётся под управление отдельному серверу.
5. **Denormalization** - техника работы с БД, обратная процессу нормализации.  
   Цель денормализации в том, чтобы путём нарушения правил(а) ACID повысить доступность системы на read-операции.
6. **SQL Tuning** - способ повышения доступности через оптимизацию SQL-запросов.
